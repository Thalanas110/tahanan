# DASS-21 Mental Monitoring Design

## Purpose

Add a couple-only Mental Monitoring area that lets each partner complete the DASS-21 questionnaire and view score trends. It is a monitoring tool, not a diagnostic tool. The application must never persist individual answers; it persists only the final Depression, Anxiety, and Stress scores needed for trend charts.

## Scope and product behavior

- Add `/mental-monitoring` to the protected application and More navigation. It is available only in the partner room, never in a Circle of Friends room.
- Show a permanent, prominent notice before a questionnaire can begin and in its questionnaire header: DASS-21 is for monitoring, not diagnosis; consult a doctor or qualified mental-health professional if signs of emotional difficulty keep recurring. The results view repeats this notice. It will also advise someone in immediate danger to contact local emergency services.
- Present the 21 statements and the supplied four-point response scale for the previous week. A respondent must answer every question before submission.
- Keep responses only in component memory while the questionnaire is open. Reset them after a successful submission and never place them in local storage, React Query data, an API request, logs, or the database.
- Calculate the three seven-item subtotals in the browser, multiply each by two, and submit only the final integer scores. Severity labels are derived from the supplied DASS-21 bands when displayed and are never stored.
- Both members of the partner couple may view each other's score history. The dashboard provides separate `You` and `Partner` views, each with three time-series lines for Depression, Anxiety, and Stress.

## DASS-21 scoring

The source of truth for statement text, item groups, scale wording, and bands is `docs/dass-21/DASS-21.md`.

| Scale | Question numbers | Final score |
| --- | --- | --- |
| Depression | 3, 5, 10, 13, 16, 17, 21 | Sum of responses x 2 |
| Anxiety | 2, 4, 7, 9, 15, 19, 20 | Sum of responses x 2 |
| Stress | 1, 6, 8, 11, 12, 14, 18 | Sum of responses x 2 |

The application exposes DASS-21 conventional range labels only as score context, explicitly not as a diagnosis.

## Data model and access boundaries

A new migration creates `dass_monitoring_entries` with:

- `id` UUID primary key generated by the server.
- `couple_id` and `submitted_by`, both non-null foreign keys; no `cof_id` column.
- `ciphertext`, `ciphertext_iv`, `encrypted_data_key`, `encrypted_data_key_iv`, and `key_version`, all non-null metadata required for decryption and key rotation.
- `created_at` timestamp.

The encrypted payload is a versioned object containing only `depression`, `anxiety`, and `stress` final scores. It excludes answers, labels, free text, and derived chart fields. Encryption uses authenticated additional data containing the record ID, couple ID, submitting user ID, schema version, and key version, preventing a ciphertext from being moved between records or couples.

RLS is enabled with no direct client policies. The browser cannot select, insert, update, or delete this table through the Data API. Only dedicated Edge Functions use a privileged server client after explicit caller and couple-membership checks.

## Envelope encryption and automatic recovery

Each entry uses envelope encryption rather than double-encrypting the score payload:

1. The write function validates the authenticated caller, couple membership, and final score shape (integer, even, and 0 through 42).
2. It generates a cryptographically random 256-bit data-encryption key (DEK) and 96-bit IV.
3. It encrypts the score payload once using AES-256-GCM and the DEK.
4. It retrieves the current 256-bit key-encryption key (KEK) from a named Supabase Vault secret, held outside frontend code and application `.env` files.
5. It encrypts the DEK with AES-256-GCM and the KEK, using a distinct random 96-bit IV.
6. It inserts only the encrypted payload, encrypted DEK, IVs, key version, ownership metadata, and timestamp.

Supabase Vault stores the KEK encrypted at rest under a project-managed root key that is separate from application data. The migration generates the KEK inside the database and registers it in Vault, so the raw production key is neither committed nor configured in a frontend or application environment file. Access to Vault's decrypted view is restricted to a narrowly scoped security-definer database function callable only by the Edge Function's server role.

For reads, the history function validates membership, uses the KEK to unwrap the DEK, decrypts the score payload in function memory, and returns only the chart-ready score records over the authenticated TLS connection. This allows either authenticated partner to access history on a new device. It is managed encryption at rest, not end-to-end encryption: the trusted server can decrypt for authorized users.

Key rotation creates a new named Vault secret and a new migration. New entries use the new `key_version`; historical entries continue to be decrypted with their recorded version until an explicit re-encryption migration is planned.

## Frontend architecture

`src/lib/dass21.ts` contains immutable questionnaire metadata, scoring, final-score validation, severity-range derivation, and no persistence logic. `src/lib/dassEncryption.ts` or an Edge shared helper contains the portable AES-GCM envelope operations; it takes key bytes as arguments and never imports a key from browser configuration.

`useDassMonitoring` communicates only with `create-dass-monitoring-entry` and `get-dass-monitoring-history` Edge Functions. It invalidates the selected couple's history after a successful submit. It receives and retains only score records, never answers.

`mental-monitoring.tsx` renders the introductory notice, questionnaire state, result summary, empty state, and responsive Recharts trend dashboard. Its logic hook owns the ephemeral response state and clears it after successful persistence or deliberate cancellation.

## Error handling

- Block submission until all 21 responses are valid values from zero through three.
- Surface a neutral retry message for network, encryption, or save failures; retain in-memory responses after a failed save so the user can retry, but never cache them durably.
- Reject malformed, out-of-range, odd, or unauthorized scores in the write function without inserting anything.
- The history function returns an authorization error for non-members and a generic unavailable message if a ciphertext cannot be authenticated or decrypted. It does not expose key or cryptographic implementation details to the browser.
- Display an understandable empty state when a selected partner has not completed an assessment.

## Verification

- Unit-test the exact question groupings, multiplication, boundary validation, and all conventional severity bands.
- Unit-test AES-256-GCM envelope round trips and assert that modified ciphertext, IV, additional authenticated data, or wrapped DEKs fail authentication.
- Unit-test that the write payload contains scores only and no response array or question text.
- Type-check and production-build the application.
- Inspect the migration for no plaintext score columns, no answer columns, RLS enabled, and no direct client policy.

## Non-goals

- Diagnosis, clinical triage, treatment advice, or alerts based on scores.
- Partner or Circle of Friends access to questionnaire answers.
- End-to-end encryption without server recovery.
- Offline persistence of questionnaire responses.
